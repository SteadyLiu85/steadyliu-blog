**Processor Architecture**

**处理器体系结构：从逻辑门到执行流**

这章的主要内容为一个简化的指令集 Y86-64 的设计以及 Y86-64 处理器的实现（顺序实现和 pipeline 实现）。由于Y86本身不具有普适性，本篇只讨论其对底层开发有实际指导意义的硬件逻辑。

## 一、 指令集架构 (ISA) 设计

### 1. 指令编码

一条机器指令在内存中仅仅是一串连续的字节。硬件必须从这串字节中极其高效地解析出控制信号。在精简的架构设计中，指令通常被拆分为以下几个信息段：

- **指令标识 (Opcode & Function)：** 通常占用 1 个字节。高 4 位为 Code（大类，如算术运算、条件跳转），低 4 位为 Function（细分，如具体的加法或减法、大于或等于）。

  

- **寄存器指示符 (Register Specifier)：** 占用 1 个字节。由于通用寄存器数量通常有限（例如 15 个），只需要 4 个 bit 即可唯一标识一个寄存器。高 4 位指定源寄存器 $rA$，低 4 位指定目的寄存器 $rB$。若无需寄存器参与，则填入占位符（如 `0xF`）。

  

- **常数 / 偏移量 (Constant Word)：** 占用 8 个字节。用于存放立即数或内存绝对地址，按小端法 (Little Endian) 排列。

### 2. 操作的拆分

真实的 x86 架构中，一条 `movq` 指令可以处理无数种寻址组合。而在更偏向 RISC（精简指令集）的设计中，内存访问和计算被严格解耦。

例如，数据传送被强制拆分为四个独立指令：`irmovq`（立即数到寄存器）、`rrmovq`（寄存器之间）、`mrmovq`（内存到寄存器）、`rmmovq`（寄存器到内存）。算术指令（如 `addq`）被限制为只能在寄存器之间进行，不允许直接对内存操作。

> 我们最初看汇编会觉得这种限制很繁琐。但在硬件层面，统一的位宽和确定的操作数来源，意味着 CPU 在**译码 (Decode)** 阶段不需要做复杂的条件分支判断，硬件布线可以极其规整。精简指令集（RISC）之所以能支撑起现代极深的高频流水线，正是因为这种对硬件友好的特性。

### 3. 硬件级的异常响应 (Exceptions)

CPU 内部维护着一个状态码（Stat），用于监控机器的健康度。当遇到以下情况时，处理器会阻断正常的取指流程：

- **HLT：** 执行了主动停机指令。
- **ADR (Address Error)：** 内存越界，试图访问非法的物理地址空间。
- **INS (Illegal Instruction)：** 取指阶段读到了硬件无法解析的操作码。

> 在 Linux 系统下进行 C/C++ 开发时，最让人头疼的 `Segmentation Fault` (段错误) 或 `Illegal Instruction` (非法指令) 崩溃，其物理层面正是 CPU 硬件触发了 `ADR` 或 `INS` 状态。硬件一旦捕获这些异常，就会通过中断向量表将控制权强制交还给操作系统的内核态异常处理程序，最终由 OS 给我们的进程发送 `SIGSEGV` 或 `SIGILL` 信号并生成 Coredump。

## 二、 硬件控制基础：组合逻辑与时序逻辑

要理解 CPU 的执行流，必须先建立两种截然不同的电路时间观念。

### 1. 组合逻辑 (Combinational Logic)

由基础的与、或、非门组成，负责执行所有纯粹的数据加工计算（如 ALU、多路复用器 MUX）。

- **本质：** 无状态、无记忆。它就像程序中的纯函数。
- **工作模式：** 输出仅仅是当前输入引脚电平的实时反映。只要输入一变，经过极短的物理门延迟（逻辑延迟）后，输出必定改变。
- **多路复用器 (MUX)：** 它是硬件世界的 `switch-case`。它接收多路数据输入和一组控制信号，根据控制信号的求值结果，连通某一路数据到输出端。在硬件控制语言 (HCL) 中，通常用自上而下的条件表达式来描述。

### 2. 时序逻辑 (Sequential Logic)

时序电路是 CPU 拥有“记忆”和维持“节拍”的关键，典型代表是寄存器 (Register) 和内存 (Memory)。

- **时钟 (Clock)：** 统领整个 CPU 运转的全局震荡信号。
- **读写的根本差异：** 硬件的**读取是实时的组合逻辑**（地址线电平一稳定，数据线上立刻出现值）；但**写入必须受到时钟控制**。哪怕输入引脚的数据已经算好并稳定存在，存储介质也绝不会发生状态翻转，唯有当全局时钟信号的**上升沿 (Clock Rise)** 到来时，数据才会被瞬间“拍”进存储器中。

## 三、 SEQ (顺序) 处理器的六阶段生命周期

SEQ（Sequential）是一种让每条指令在一个超长的时钟周期内，完整走完所有流程的直观架构。这也是构建流水线 (Pipeline) 的先决条件。

### 1. 取指 (Fetch)

CPU 将程序计数器 (PC) 的值作为内存地址，从指令内存中读取若干字节。

- 提取出指令指示符（Code 和 Function）。
- 提取出寄存器指示符（$rA$ 和 $rB$）。
- 提取出 8 字节的常数 / 地址字（$valC$）。
- 计算出当前指令的总长度，得出下一条指令的顺序地址（$valP$）。

### 2. 译码 (Decode)

根据取指阶段解析出的寄存器 ID（$rA$ 和 $rB$），直接从寄存器文件 (Register File) 中读出对应的操作数，通常记为 $valA$ 和 $valB$。

- *注：这里的“译码”命名有些诡异，其实指令类型的解析在 Fetch 阶段就已完成，Decode 阶段硬件实际做的工作是“读取寄存器”。*

### 3. 执行 (Execute)

这是最耗时的算术逻辑阶段。算术逻辑单元 (ALU) 接收 $valA$、$valB$ 或 $valC$，根据指令类型执行具体的操作：

- 如果是算术指令，执行加减并更新状态标志位（Condition Codes）。

  

- 如果是访存指令，将基址和偏移量相加，计算出有效的内存物理地址。

  

- 如果是跳转指令，结合状态标志位，计算出分支预测的结果（是否应该跳转）。ALU 最终输出的计算结果记为 $valE$。

### 4. 访存 (Memory)

主要针对读写内存的指令。

- 如果是写内存指令，将数据写入执行阶段算出的有效地址中。
- 如果是读内存指令，从有效地址中读出数据，记为 $valM$。

### 5. 写回 (Write back)

将 ALU 的计算结果 ($valE$) 或内存读出的数据 ($valM$)，写回到目的寄存器（如 $rA$ 或 $rB$）中。

### 6. 更新 PC (PC update)

决定下一条将要执行的指令地址。如果没有发生控制流跳转，PC 更新为 Fetch 阶段算出的 $valP$；如果发生了分支跳转（如 `call` 或条件 `jmp`），PC 更新为跳转的目标地址 $valC$ 或从栈中弹出的返回地址 $valM$。

## 四、 硬件设计：禁止回读 (No Reading Back)

**一条指令绝对不能在更新了某个状态后，立马在同一个周期内读取这个新状态**。

以 `pushq %rsp` 这个复杂的边界操作为例。它不仅要将 `%rsp` 的旧值写入内存，还要将 `%rsp` 自身减去 8。

- 在电路上，访存阶段（Memory）的组合逻辑会根据旧的 `%rsp` 算出写入地址，并把旧 `%rsp` 的值放到写数据线上。
- 与此同时，执行阶段（Execute）的 ALU 会算出 `%rsp - 8` 的结果 $valE$。
- 这所有的一切操作，在时钟周期内都仅仅是电信号在导线上的传播和组合。**没有任何寄存器被修改。**
- 当且仅当周期结束，**时钟上升沿到来的那一瞬间**，硬件会同时进行两个动作：把数据写进内存，并且把 $valE$ 写入 `%rsp` 寄存器。

> 硬件时序直接导致了高级流水线中无法避免的“数据冒险 (Data Hazard)”。当指令 A 算出结果但还没等到时钟沿写回寄存器时，紧随其后的指令 B 就要在译码阶段去读这个寄存器，此时 B 读到的就是错误的老数据。
>
> 现代 CPU 依靠内部复杂的数据转发 (Bypassing) 网络来缓解这个问题，而现代编译器在开启 `-O2` 或 `-O3` 优化时，也会绞尽脑汁地进行指令重排 (Instruction Scheduling)，竭力把存在数据依赖的代码行拉开距离，这就是底层体系结构对上层软件优化的倒逼。

## 五、 流水线

SEQ（顺序处理器）的瓶颈：**时钟周期必须迁就最慢的那条指令**。为了等待指令走完所有的六个阶段，硬件的大部分逻辑门在绝大多数时间里都是闲置的。

现代 CPU 解决这个问题的方案是**流水线 (Pipeline)**。

它在六个执行阶段之间插入了**流水线寄存器 (Pipeline Registers)**。这样一来，当第一条指令完成“取指”进入“译码”阶段时，第二条指令就可以立刻进入“取指”阶段。理想情况下，流水线可以做到每个时钟周期都“吐出”一条执行完毕的指令，吞吐量 (Throughput) 呈指数级增长。

## 六、 冒险 (Hazards) 与气泡 (Bubbles)

但现实中的代码上下文是充满依赖的。当多条指令同时挤在流水线里时，就会发生**冒险 (Hazards)**。

### 1. 数据冒险 (Data Hazards)

下一条指令的输入，依赖于上一条指令的输出，但上一条指令还没来得及把结果写回寄存器。

- **硬件的补救（数据转发/Bypassing）：** 现代 CPU 会在内部建立复杂的旁路网络，直接把 ALU 刚算出来的结果拦截下来，横向转发给下一条正在译码的指令。
- **无法转发的死局（Load/Use 冒险）：** 如果上一条指令是读内存（直到访存阶段才能拿到数据），而下一条紧挨着的指令马上就要参与 ALU 计算，转发也来不及了。此时硬件只能强行插入一个**气泡 (Bubble，也就是 Stall 停顿)**，让流水线干等一个周期。

### 2. 控制冒险 (Control Hazards)

当遇到 `jmp` 或 `je` 等条件跳转指令时，CPU 在取下一条指令时，根本还没算出来到底该不该跳。

- **分支预测 (Branch Prediction)：** CPU 只能靠猜（比如无脑猜循环会继续跳，或者按历史记录猜）。如果猜对了，流水线丝滑运转。
- **预测失败的惩罚：** 如果猜错了，CPU 必须立刻把流水线里错误取出的指令全部作废（丢弃），这会产生巨量的气泡，损失几十个时钟周期。

> 1. **循环展开 (Loop Unrolling)**：不仅减少了循环变量递增的开销，更重要的是它拉开了访存指令和计算指令的距离，完美消除了 Load/Use 冒险产生的气泡。
> 2. **消除分支代码**：在 C++20 中我们广泛使用 `[[likely]]` 属性，或者把高频且难以预测的 `if-else` 改写为位运算或三目运算符（诱导编译器生成 `cmov` 条件传送指令），本质上都是为了**不让分支预测失败来破坏流水线**。

## 七、 终极性能指标：CPI 的计算

在完美的流水线中，每个时钟周期都能执行完一条指令，此时 $CPI = 1.0$。

但由于上述各种“冒险”引发了气泡，实际的 CPI 永远大于 1。它的严格数学定义如下：

$$CPI = \frac{\text{总时钟周期数}}{\text{执行的指令总数}} = \frac{\text{指令总数} + \text{气泡总数}}{\text{指令总数}} = 1.0 + \frac{\text{气泡总数}}{\text{指令总数}}$$

在具体的架构分析中，超出的部分被拆解为三种具体的惩罚因子 (Penalties)：

$$CPI = 1.0 + lp + mp + rp$$

- **$lp$ (Load Penalty)：** 加载/使用冒险带来的惩罚。代码里指针解引用和数组读取越密集，且紧接着进行计算，$lp$ 就越高。
- **$mp$ (Mispredicted Branch Penalty)：** 分支预测失败带来的惩罚。条件分支越不可预测（比如对完全无序的数组进行一半大一半小的 `if` 判断），$mp$ 飙升得越厉害。
- **$rp$ (Return Penalty)：** `ret` 指令带来的惩罚。因为函数返回地址要从内存栈里读，流水线不可避免要停顿。
