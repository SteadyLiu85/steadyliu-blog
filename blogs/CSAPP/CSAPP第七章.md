**Linking**

**链接**

这章的主要内容为程序的链接。学习链接有助于：理解链接报错，避免链接相关的 bug，理解变量（函数）的作用域，理解程序运行过程中与链接相关的步骤，了解如何使用共享库（动态链接库）。

## 一、 编译驱动程序的全链路

当你执行 `gcc -O2 -g main.c -o main` 时，GCC 驱动程序实际上在后台拉起了四个独立的工具链：

1. **预处理器 (cpp - C Preprocessor)：**
   - **动作：** 处理所有以 `#` 开头的指令（展开宏、插入 `#include` 的头文件内容、剔除注释）。
   - **产物：** `main.i`（纯 C 代码文本）。
2. **编译器 (cc1 - C Compiler)：**
   - **动作：** 进行词法、语法、语义分析和高度优化，将 C 代码降维成特定的汇编指令集。
   - **产物：** `main.s`（汇编代码文本）。
3. **汇编器 (as - Assembler)：**
   - **动作：** 将汇编指令逐行翻译成机器可执行的二进制指令，并打包成特定的文件格式。
   - **产物：** `main.o`（**可重定位目标文件**，地址尚未确定）。
4. **链接器 (ld - Linker)：**
   - **动作：** 执行**符号解析 (Symbol Resolution)** 和**重定位 (Relocation)**，将多个 `.o` 文件和系统库拼接。
   - **产物：** `main`（**可执行目标文件**，可以直接被 OS 加载器映射到物理内存运行）。

## 二、 目标文件与 ELF 

在现代 x86-64 Linux 系统中，所有的目标文件都严格遵循 **ELF (Executable and Linkable Format)** 规范。它分为三种：

1. **可重定位目标文件 (`.o`)：** 可以与其他 `.o` 文件合并。
2. **可执行目标文件 (`a.out` / `main`)：** 已经被完全重定位，可直接映射到内存。
3. **共享目标文件 (`.so`)：** 特殊的可重定位文件，支持在加载或运行时被动态加载进内存并链接。

### ELF 核心节 (Sections) 的微观意义

- **`.text`：** 存放已编译的机器代码。

- **`.rodata`：** 只读数据（如 `printf` 的格式化字符串、`switch` 语句的跳转表）。

- **`.data`：** 存放**已初始化**且初值非 0 的全局变量和静态 (`static`) 变量。

- **`.bss`：** 存放**未初始化**或初始化为 0 的全局变量和静态变量。

  →为什么不和 `.data` 合并？因为如果定义了 `int arr[100000];` 且全为 0，把它存进物理磁盘文件简直浪费。在 ELF 文件中，`.bss` 不占用任何磁盘空间，它仅仅记录一个 `size`。当 OS 加载程序时，才会在内存中开辟这块空间并用零填充（Better Save Space）。

- **`.symtab`：** 符号表。存放函数和全局变量的名称及信息（注意：它不包含局部变量，局部变量完全在栈上）。

- **`.rel.text` 与 `.rel.data`：** 重定位条目。记录了当前文件里有哪些“不知道绝对地址”的外部函数调用和全局变量引用。

## 三、 符号解析 

为代码中每一个**符号引用 (Reference)** 找到唯一对应的**符号定义 (Definition)**。

### 1. 符号的分类

- **全局符号 (Global)：** 由当前模块定义，并能被其他模块引用的符号（非 `static` 的函数和全局 C 变量）。
- **外部符号 (External)：** 在当前模块引用，但在其他模块定义的符号（通常用 `extern` 声明）。
- **本地符号 (Local)：** 带有 `static` 修饰的函数和全局变量。它们完全被封装在当前模块内部，哪怕其他模块有同名符号也不会冲突。（**C 语言的 `static` 关键字是实现模块级封装的唯一底层手段**）。

### 2. 多重定义决议

当多个文件定义了同名全局符号时，链接器依靠强弱进行决议：

- **强符号：** 函数名、**已初始化**的全局变量。
- **弱符号：** **未初始化**的全局变量。

**决议三定律：**

1. 不允许存在多个同名的强符号。**（立刻报错：`multiple definition`）**
2. 如果存在一个强符号和多个同名弱符号，选择强符号。
3. 如果存在多个同名的弱符号，链接器会**随机**挑选一个（或者挑选占用空间 `size` 最大的那个）。

> GCC 10 的破坏性更新与 `-fno-common`**
>
> 过去几十年的 C 语言习惯中，很多老式代码会在 `.h` 头文件里直接写 `int global_var;`，然后被多个 `.c` 文件 `include`。这会产生多个同名的**弱符号**。旧版 GCC 默认开启 `-fcommon` 选项，链接器会将它们悄悄合并，分配到名为 `COMMON` 的伪节中。
>
> 这种行为极易引发难以察觉的内存越界覆写 Bug。因此，**从 GCC 10 开始，默认变成了 `-fno-common`**。编译器会强制把未初始化的全局变量放进 `.bss` 段，并将其标记为**强符号**。这导致许多老项目在升级编译器后，瞬间爆发为 `multiple definition` 报错。
>
> **正确的 C/C++ 规范：** 在头文件中只使用 `extern int global_var;`（声明，不分配内存），然后在且仅在一个 `.cpp` 或 `.c` 文件中写 `int global_var = 0;`（定义）。

### 3. C++ 的 Name Mangling (名字修饰)

为什么 C 语言不支持函数重载，而 C++ 支持？

因为 C++ 编译器在生成 `.o` 文件的符号表时，会进行 Name Mangling。它将函数名、命名空间、参数类型全部压缩成一个恶心的长字符串。比如 `void foo(int)` 会被修饰成 `_Z3fooi`，而 `void foo(double)` 会变成 `_Z3food`。在链接器看来，这完全是两个毫不相干的强符号。

**所以，当 C++ 代码需要调用纯 C 语言编译的库（其符号未被修饰）时，必须加上 `extern "C"`，否则链接器找不到符号。**

## 四、 静态库 (.a) 

### 链接器的内部算法：$E$、$U$、$D$ 集合

链接器从左到右解析命令行上的文件，它在内部维护三个动态集合：

- $E$：将被合并到可执行文件的 `.o` 文件集合。
- $U$：当前未解析的符号集合（引用了但还没找到定义的符号）。
- $D$：当前已被定义的符号集合。

**算法逻辑：**

1. 遇到普通的 `.o` 文件：无条件加入 $E$。把它的未解析符号丢进 $U$，定义的符号丢进 $D$。
2. **遇到静态库 `.a` 文件：** 链接器会扫描库内的每一个 `.o` 文件，**只有当这个 `.o` 文件能解析 $U$ 集合中现存的某个符号时**，才把它提取出来加入 $E$，并更新 $U$ 和 $D$。提取完立刻扔掉这个 `.a` 库，继续向右扫描。

> **静态库依赖顺序的底层逻辑**
>
> 假设你写了 `gcc -lmath main.c`。
>
> 链接器首先遇到 `libmath.a`，此时 $U$ 集合是空的（因为 `main.c` 还没被解析）。链接器觉得库里没有任何有用的东西，直接扔掉。接着解析 `main.c`，遇到了对 `sin()` 的调用，把 `sin` 放入 $U$ 集合。然而，由于链接器**永远不会回头**，最终在 $U$ 集合中留下了未解析的 `sin`，轰然报错 `undefined reference to sin`。
>
> **解决：** 永远把依赖别人的文件放左边，被依赖的库放右边。如果有交叉依赖（A 依赖 B，B 又依赖 A），可以通过强制重复写来破解：`gcc main.c -lA -lB -lA`。

## 五、 重定位 (Relocation)

当所有被选中的 `.o` 文件聚集完毕，链接器开始进行**重定位**。它将所有 `.text` 节合并成一个巨大的新 `.text` 节，给每一条汇编指令和全局变量分配最终的运行时内存绝对地址。

接着，它需要通过 `.rel.text` 中记录的条目，去修改代码中的地址占位符。CSAPP 重点讲解了两种 x86-64 的重定位类型：

### 1. `R_X86_64_32` (绝对重定位)

直接把占位符改成目标符号在内存中的绝对地址（通常用于引用全局数据）。

- 公式：`真实地址 = ADDR(symbol) + addend`（`addend` 是一个常数偏移，用于处理如 `&array[2]` 这样的寻址）。

### 2. `R_X86_64_PC32` (相对重定位 / PC 相对寻址)

现代 CPU 的 `call` 或 `jmp` 指令为了极致的紧凑性，通常使用相对当前程序计数器 (PC) 的偏移量跳转。

- 公式：`补丁值 = ADDR(symbol) + addend - 引用处的PC地址`。

  当 CPU 执行到这条指令时，硬件会自动把当前的 PC 值加上这个补丁值，落在目标函数的绝对地址上。

## 六、 动态链接 (.so) 与位置无关代码 (PIC)

静态库会导致每一个程序都在磁盘和内存中塞满相同的代码（如 `printf`）。

**共享库 (Shared Library, .so)** 颠覆了这一点：它在磁盘上只有一个文件，在物理内存中也只有一份代码段（`.text`），被无数个进程共享。

**悖论：** 既然代码段是多进程共享的，它就必须被设为**只读**。既然是只读的，动态链接器在程序加载时，就绝对不可能去修改代码段里的相对地址（不能打重定位补丁）。那共享库如何正确调用外部函数？

**解法：PIC (Position-Independent Code，位置无关代码)。**

### GOT 与 PLT

**无论共享库被加载到内存的哪里，它的代码段和数据段之间的相对距离是永远固定的。**

1. **GOT (Global Offset Table, 全局偏移表)：**

   建在**数据段（可读写，每个进程私有一份）**里的一张表，里面存放外部函数的绝对地址。代码段通过不变的相对距离，去读 GOT 表里的数据，从而实现间接跳转。

2. **PLT (Procedure Linkage Table, 过程链接表) 与 延迟绑定：**

   如果一个 `.so` 依赖了一万个外部函数，程序启动时要把 GOT 全填满，启动耗时会非常恐怖。因此系统引入了 PLT（位于代码段的一排跳板指令）来实现**延迟绑定 (Lazy Binding)**。

**PLT/GOT 的执行流（例如调用 `addvec`）：**

- **第一次调用：**
  1. 代码 `call addvec@plt`。
  2. `PLT[2]` 里的指令执行：无脑跳转到 `GOT[4]` 中存放的地址。
  3. **重点：** 此时 `GOT[4]` 尚未被解析，它里面预先存放的地址，正好指向 `PLT[2]` 的下一条指令。
  4. 继续执行 `PLT[2]`：往栈里压入当前函数的 ID，然后跳到 `PLT[0]`。
  5. `PLT[0]` 跳转进入**动态链接器 (`ld-linux.so`)**。
  6. 动态链接器查表、寻找真正的 `addvec` 地址，将其覆盖写回 `GOT[4]`，并跳转过去执行。
- **第二次及以后的调用：**
  1. 代码 `call addvec@plt`。
  2. `PLT[2]` 无脑跳转到 `GOT[4]`。
  3. 此时 `GOT[4]` 里已经是真实的地址，瞬间完成跳转，不再经过动态链接器。性能几乎无损耗。

## 七、 库打桩 (Library Interpositioning)

### 1. 运行时加载 

C/C++ 允许你在程序运行到一半时，手动唤醒动态链接器去加载某个 `.so`。这广泛用于各种高性能插件系统。

通过引入 `<dlfcn.h>`：

- `dlopen("libfoo.so", RTLD_LAZY)`：加载库。
- `dlsym(handle, "my_func")`：通过字符串名称查找函数真实的内存地址。
- `dlclose(handle)`：卸载库。

### 2. 库打桩 

你可以强制系统优先使用你写的函数，而不是系统库的同名函数（比如劫持 `malloc`）。

- **编译时打桩：** 修改源代码，利用 `#define malloc my_malloc` 偷梁换柱。
- **链接时打桩：** 给 `ld` 传递 `--wrap=malloc` 参数。链接器会自动把对 `malloc` 的调用重定向到你写的 `__wrap_malloc`。
- **运行时打桩 ：** 甚至不需要重新编译目标程序。只需在运行前设置环境变量 `LD_PRELOAD="/path/to/my_hack.so"`。动态链接器会**无视一切规则，强制最高优先级**加载你的 `.so`。只要里面有和系统同名的函数，系统调用将直接被劫持。
