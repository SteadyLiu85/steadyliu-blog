**Optimizing Program Performance**

**优化程序性能**

这章的主要内容为如何通过了解CPU架构来优化程序的性能

## 一、 现代处理器微架构 

要榨干性能，必须先理解 CPU 是如何执行代码的：

- **超标量 (Superscalar)：** CPU 内部有多个功能单元（如 2 个加载单元，2 个乘法单元），在一个时钟周期内可以同时发射并执行多条指令。
- **乱序执行 (Out-of-Order Execution)：** 处理器会打破代码的字面顺序，只要指令之间没有数据依赖，它就会提前把后面的指令拉进来执行。
- **微指令转化：** 指令译码器 (Instruction Decoder) 会把复杂的 x86 汇编指令进一步切碎成极简的微操作 ($\mu$-ops) 交给执行单元。
- **退役单元 (Retirement Unit)：** 负责兜底，确保乱序执行的结果最终能按照程序原本的顺序提交，维持状态一致性（这也是分支预测失败时能够安全回滚的保障）。

------

## 二、 程序性能的度量

- **CPE (Cycles Per Element)：** 处理每个元素所需的时钟周期数。这是衡量循环性能最直观的绝对指标，CPE 越低，性能越强。
- **延迟 (Latency)：** 完成单条指令所需的总周期数（比如一次浮点乘法可能需要 3 个周期）。
- **发射时间 (Issue Time)：** 连续执行两条同类指令之间的最小周期数。现代 CPU 大多是完全流水线化的，发射时间通常为 1。
- **容量 (Capacity)：** 执行某种运算的独立硬件单元数量。

## 三、 编译器盲区与代码重构

### 1. 不必要的过程调用 (Code Motion)

千万不要在循环条件里放置黑盒函数或包含复杂计算的函数（如 `for (int i = 0; i < strlen(s); i++)`）。编译器不敢把 `strlen` 挪到外面，因为它怕函数内部有隐藏的副作用。

**对策：** 执行**代码外提**，将与循环迭代无关的计算提取到循环外部的局部变量中。

### 2. 内存别名 (Memory Aliasing)

当两个指针可能指向同一块内存时，编译器会放弃所有的寄存器缓存优化，采取最保守的每次读写内存策略。

> 对于内存别名**
>
> **反例：**
>
> ```
>void add_array(long *src, long *dest, int n) {
>  for (int i = 0; i < n; i++) {
>      *dest = *dest + src[i]; 
>     }
>    }
>    ```
> 
> *底层的无奈：* 编译器怕 `dest` 和 `src[i]` 指向同一个地址。因此在汇编层，每次循环 CPU 都要执行：读 `src` -> 读 `dest` -> 相加 -> 写回 `dest`。极其浪费访存带宽。
>
> **优化后：**
>
> ```
>void add_array_optimized(long *src, long *dest, int n) {
>  long acc = 0; // 强制使用寄存器
> for (int i = 0; i < n; i++) {
>      acc = acc + src[i];
>  }
>     *dest = acc; // 循环结束后再写回
>    }
>    ```
>    
>    `acc` 必定分配在寄存器中。整个循环内部没有任何内存写操作，性能提升很大。

## 四、 突破性能界限：指令级并行 (ILP)

即使排除了内存干扰，程序的性能依然受制于两个物理界限：

- **延迟界限 (Latency Bound)：** 由指令间的数据依赖决定，指令必须严格串行执行。
- **吞吐量界限 (Throughput Bound)：** 由硬件功能单元的数量决定。

为了突破延迟界限，逼近吞吐量界限，我们需要进行循环级别的改造。

### 1. 循环展开 (Loop Unrolling, $k \times 1$)

一次迭代处理 $k$ 个元素。它不仅能减少索引自增和条件判断的控制开销，更重要的是为后续的指令调度腾出了施展空间。

### 2. 多路累加 ($k \times k$ 展开)

仅仅展开不够，如果只有一条依赖链（如 `acc = acc + data[i]`），CPU 的多个计算单元依然在闲置。通过引入多个独立的累加器（`acc0`, `acc1`），彻底打碎数据相关性链条。

### 3. 重新结合变换 

> 重新结合的示例：
>
> **反例：**
>
> ```
>acc = (acc * data[i]) * data[i+1];
> ```
> 
> *依赖链：* 算完 `acc * data[i]`，才能算外层乘法。两条乘法指令是前后串行的。
>
> **优化后：**
>
> ```
>acc = acc * (data[i] * data[i+1]);
> ```
>
> 仅仅换了括号的位置，由于 `data[i] * data[i+1]` 不依赖当前的 `acc`，现代 CPU 会立刻在一个乘法单元里算这个括号，同时在另一个乘法单元里处理上一次的 `acc`，CPE 直接减半

### 4. 浮点数无结合性

为什么开启了 `-O3` 优化，编译器还是不自动加这个括号？

因为对于**浮点数 (Float/Double)**，舍入误差会导致 $(a+b)+c \neq a+(b+c)$。严格遵循 IEEE 754 标准的编译器绝对不敢擅自改变浮点数的计算顺序。必须显式开启 `-ffast-math` 编译选项。

## 五、 影响流水线的瓶颈

- **寄存器溢出 (Register Pressure)：** 如果写了 $20 \times 20$ 的循环展开，使用的并行累加变量超过了 CPU 物理寄存器的数量（x86-64 有 16 个通用寄存器），多出来的变量会被“溢出”到内存栈上，性能反而会崩掉。
- **分支预测处罚 (Misprediction Penalty)：** 现代 CPU 极其害怕无法预测的 `if-else`。预测失败会导致整条流水线清空，损失几十个周期。
  - **对策：** 用三目运算符或位掩码诱导编译器生成 `cmov`（条件传送指令）来消灭跳转；或者通过**循环拆分 (Loop Unswitching)** 将判断移出循环主体。
- **SIMD (Vectorization)：** 利用 AVX 向量寄存器，一条指令同时操作多个数据。

## 六、 Amdahl定律

$$S = \frac{1}{(1-\alpha) + \frac{\alpha}{k}}$$

这是性能优化的最高指导法则。

如果你把一个只占总运行时间 10% 的函数优化到了光速（提升 $k \to \infty$），你的整体程序最多也只能快 1.11 倍。因此，永远不要凭直觉去盲目优化。
