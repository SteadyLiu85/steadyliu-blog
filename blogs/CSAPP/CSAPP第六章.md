**The Memory Hierarchy**

**存储器层次结构**

这章的主要内容有：各种存储设备（RAM、ROM、HDD、SSD）的特点、程序的局部性、缓存的结构以及原理、缓存对程序性能的影响。

## 一、 存储器

### 1. 随机访问存储器 (RAM)

- **SRAM (静态 RAM)：** 每个 bit 由 6 个晶体管组成。只要不断电，状态就极其稳定。**速度极快，造价极高，功耗极大。** 专用于 CPU 内部的 L1/L2/L3 Cache。
- **DRAM (动态 RAM)：** 每个 bit 仅用 1 个微小电容的电荷来存储。**高密度，造价低。** 缺点是电容会漏电，必须由内存控制器周期性地读取并重写（动态刷新）。它构成了主存 (Main Memory)。

### 2. 非易失性存储与 DMA 机制

- **机械硬盘 (HDD)：** 依靠主轴旋转和磁头寻道。一次扇区读取的绝对大头耗在了**寻道时间 (Seek)** 和**等待旋转 (Rotation)** 上。
- **固态硬盘 (SSD)：** 基于 Flash 内存。数据按页 (Page) 读写，但写入前必须按极大的块 (Block) 进行物理擦除。擦除极度耗时且有寿命限制，因此固件层必须做**磨损均衡 (Wear-leveling)**。
- **DMA (直接内存访问)：** 磁盘读写太慢，如果让 CPU 傻等就太浪费了。现代系统中，CPU 只需要给磁盘控制器下发一条指令，然后就可以去执行其他进程。磁盘读完数据后，会**直接通过系统总线把数据塞进主存**，最后发一个硬件中断通知 CPU。

## 二、 局部性原理 (Locality)

1. **时间局部性 (Temporal Locality)：** 刚被访问过的数据，极有可能在短期内再次被访问（例如循环内部的累加器变量 `acc`）。
2. **空间局部性 (Spatial Locality)：** 刚被访问过的数据，其物理内存地址相邻的数据，极有可能很快被访问。

> 在算法复杂度上，`std::list`（链表）的中间插入是 $O(1)$，似乎优于 `std::vector` 的 $O(N)$。但在真实的物理机器上，链表节点是通过 `new` 零散分配在堆内存中的，空间局部性极差，遍历时会疯狂触发 Cache Miss。而 `vector` 底层是一整块连续物理内存，完美契合 CPU 硬件的**按块预取 (Prefetch)** 机制。在海量数据的遍历和轻度插入场景下，`vector` 的绝对物理耗时通常会对远少于`list` 。

## 三、 Cache 

Cache 是透明的黑盒，但硬件是通过对物理地址的**位切片（Bit Slicing）**来实现极其暴力的 $O(1)$ 硬件级哈希查找的。

### 1. (S, E, B) 核心参数

一个 Cache 的总容量 $C = S \times E \times B$。

- **$S$ (Cache Sets)：** 组数，通常是 $2$ 的幂（$S = 2^s$）。
- **$E$ (Lines per Set)：** 每个组的行数（路数）。$E=1$ 是直接映射，$E>1$ 是组相联（现代 CPU 多为 8-way 或 16-way），如果只有一个组且包含所有行，叫全相联。
- **$B$ (Block Size)：** 每个 Cache Line 装载的数据块大小，通常为 64 字节（$B = 2^b$）。

### 2. 物理地址的组成

假设系统物理地址位宽为 $m$ 位。当 CPU 拿到一个地址去查 Cache 时，硬件会瞬间将这 $m$ 个 bit 切成三段：

```
[ t 位的 Tag (标记) | s 位的 Set Index (组索引) | b 位的 Block Offset (块偏移) ]
```

- **第一步 (找组)：** 抽取中间的 $s$ 位，直接定位到具体的 Cache Set。
- **第二步 (比对)：** 遍历该 Set 中的 $E$ 行，检查有效位 (Valid Bit) 是否为 1，并且比对高位 $t$ 组成的 Tag 是否与当前地址的 Tag 匹配。
- **第三步 (提数据)：** 如果命中，根据低位 $b$ 提供的偏移量，从 64 字节的 Block 中把真正需要的几个字节（如一个 `int`）抠出来给 CPU。

> 如果用地址的**高位**做组索引会怎样？连续的物理内存块（比如一个大数组）的高位通常是相同的。这会导致数组的前几个块全部被死死映射到同一个 Cache Set 里，瞬间填满并互相驱逐（Thrashing），而其他的 Set 全是空的。
>
> **使用中间的 $s$ 位做组索引**，让物理内存中相邻的数据块，被均匀、轮流地散列分布到不同的 Cache Set 中，利用了空间局部性。

## 四、 缓存没命中的情况(The 3 C's)

1. **强制性不命中 (Cold / Compulsory Miss)：** Cache 刚开机是空的，第一次访问必然不命中。这是物理规律，无法避免（除非依赖硬件的强制预取）。
2. **容量不命中 (Capacity Miss)：** 你的程序在频繁访问一个巨大的工作集（比如 100MB 的大数组），而 Cache 总共才 8MB。Cache 被撑爆，只能不断踢出旧数据。
3. **冲突不命中 (Conflict Miss)：** 缓存明明还有大量空余，但因为多个高频访问的内存块，其物理地址的低位计算出的 Set Index 恰好一模一样，导致它们在同一个组里疯狂互相驱逐。

> 在进行底层图像处理或声明矩阵大数组时，如果大小正好是精确的 $2^{10}$ 等 $2$ 的幂次方，那么它们每行首地址的低位几乎肯定是完全对齐的！交替读写这些行时，会 100% 触发冲突不命中。
>
> 故可以故意在数组每行的末尾增加几个毫无意义的 Padding（填充变量），或者刻意分配非 $2^N$ 大小的内存。这看似浪费了几个字节，实则强行错开了地址的低位索引，可彻底解决冲突不命中的问题。

## 五、 如何榨干cache

### 1. 结构体布局优化

由于 Cache Line 通常是 64 字节，一次会把连续的 64 字节搬进缓存。如果在 C/C++ 中设计结构体时随意排列变量，会导致变量跨越两个 Cache Line，或者浪费大量的硬件 Padding 空间。

**优化铁律：** 永远把结构体中的变量按照类型大小（如 `double` -> `int` -> `char`）**从大到小依次排列**，这能最大化地把有用数据塞进同一个 Cache Line 中。

### 2. 矩阵乘法的循环重排

同样是 $N \times N$ 的矩阵相乘：

- **`ijk` 顺序：** 内层循环在**按列读取**矩阵 B（`B[k][j]`，随着 `k` 递增，地址跳跃极大）。每一次读取都会精准跨过一个 Cache Block，造成极高的 Miss 率。
- **`kij` 顺序：** 仅仅交换了两个 `for` 循环的位置！此时最内层索引是 `j`，这意味着你在**严格按行顺序读取**矩阵 C 和 B。代码的时间复杂度依然是 $O(N^3)$，但由于完美迎合了 Cache 机制，实际运行时间可能会骤降十几倍。

### 3. 分块 (Blocking / Tiling)

如果矩阵实在太大，连一整行都塞不进 L1 Cache 怎么办（触发了容量不命中）？

我们需要使用**分块技术**。也就是将大矩阵在逻辑上切分成适合 L1 Cache 大小（例如 $32 \times 32$）的子矩阵块（Block）。让 CPU 集中火力在 L1 Cache 中彻底算完这个 $32 \times 32$ 小块的所有内积，然后再去加载下一个小块。

**分块技术是现今所有底层线性代数库（如 BLAS, Eigen）和深度学习算子库（如 cuBLAS）的核心。**

### 4. 多核下的伪共享 (False Sharing)

在多线程 C++ 开发中，如果线程 A 在狂刷变量 `x`，线程 B 在狂刷变量 `y`。虽然 `x` 和 `y` 毫无逻辑关联，但如果它们在内存中挨得太近，被塞进了**同一个 64 字节的 Cache Line**中。

由于现代 CPU 的缓存一致性协议（如 MESI），线程 A 修改 `x` 时，会把线程 B 所在核心的整条 Cache Line 宣告失效 (Invalidate)；线程 B 改 `y` 时又会让 A 的失效。两条线程就像在打乒乓球一样疯狂让对方的 Cache 穿透到主存。

**解决办法：** 使用 `alignas(64)` 强制让 `x` 和 `y` 各自独占一条 Cache Line
