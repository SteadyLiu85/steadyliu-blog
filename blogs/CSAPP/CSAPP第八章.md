**Exceptional Control Flow**

**异常控制流**

本章的主要内容为 exception、system call、process、signal、longjmp。

在一般情况下，PC 会按照指令的顺序以及跳转指令来变化。但在很多时候，这样的控制流是不能满足需要的，需要 *exceptional control flow* (ECF) 作为跳转指令的补充，以处理一些“异常”的或者来自“外部”的变化。

## 一、 异常 (Exceptions)

异常是硬件和操作系统内核协作的产物。当处理器检测到状态变化（Event）时，它会通过硬件层面的**异常表 (Exception Table / 中断向量表)**，根据异常号 (Exception Number) 进行索引，直接跳入操作系统内核预设的**异常处理程序 (Exception Handler)**。

### 1. 异常的分类

- **中断 (Interrupt)：** **纯异步**，来自处理器外部。例如网卡接收到数据、时钟定时器触发。处理器会在执行完当前指令后，去执行中断处理程序，完毕后**返回下一条指令**。
- **陷阱 (Trap) 与系统调用：** **同步**，是有意为之的异常。应用程序如果想读写文件、创建进程，绝不能直接操作硬件，必须主动触发陷阱，进入内核态（Kernel Mode）。处理完毕后**返回下一条指令**。
- **故障 (Fault)：** **同步**，由潜在的可恢复错误引起。例如**缺页异常 (Page Fault)**。处理器试图访问内存，发现数据还在磁盘上，于是触发故障。内核的页错误处理程序会将物理页调入内存，然后**重新执行触发故障的那条指令**。如果像段错误（SegFault）那样无法修复，直接终止程序。
- **终止 (Abort)：** **同步**，不可恢复的致命硬件错误（如 DRAM 校验和错误）。直接终止，绝不返回。

### 2. 系统调用 (System Call) 的底层

在 x86-64 Linux 中，系统调用通过 `syscall` 指令触发。

- **传参机制：** C 语言普通的函数调用前 6 个参数通过 `%rdi, %rsi, %rdx, %rcx, %r8, %r9` 传递。而 `syscall` **不同**：它使用 `%rdi, %rsi, %rdx, %r10, %r8, %r9`。
- **系统调用号：** 必须放在 `%rax` 寄存器中（例如 `read` 是 0，`write` 是 1，`fork` 是 57）。
- **返回值：** 结果会覆盖 `%rax`，如果出错，`%rax` 中会包含一个负的错误码，C 标准库的 wrapper 函数会把这个负数截获，转成正数赋值给全局变量 `errno`，然后对外返回 `-1`。

## 二、 进程 (Processes)

1. **逻辑控制流：** 每个程序都感觉自己在独占整个 CPU。
2. **私有地址空间：** 每个程序都感觉自己在独占系统中所有的物理内存。

### 1. 模式位 (Mode Bit) 

CPU 内部有一个控制寄存器，其中包含一个模式位。

- **用户态 (User Mode)：** 模式位为 0。进程不允许执行特权指令（如直接停止处理器、修改基址寄存器），不允许直接访问地址空间中内核区的代码和数据。
- **内核态 (Kernel Mode)：** 模式位为 1。**唯一从用户态进入内核态的方法，就是触发异常（如中断、故障、陷入）。** 异常发生的那一瞬间，硬件会自动将模式位设为 1，跳入内核栈执行。

### 2. 上下文切换 (Context Switch) 

内核通过调度器 (Scheduler) 实现多任务并发。上下文切换包含极其沉重的物理开销：

1. 保存当前进程的上下文（通用寄存器、浮点寄存器、PC、状态寄存器）。
2. 恢复目标进程的上下文。
3. **开销：切换页表目录指针（如 x86 的 CR3 寄存器）。** 这会直接导致 CPU 内部的 TLB（Translation Lookaside Buffer）全部失效，接下来的几十条指令会遭遇极其严重的 Cache Miss 和内存翻译延迟。

## 三、 进程控制 API

### 1. `fork()`：写时复制 (COW) 

`fork()` 调用一次，返回两次。子进程获得与父进程完全相同但独立的用户级虚拟地址空间副本。

> **写时复制 (Copy-On-Write)**
>
> 早期 Unix 的 `fork` 会真的把物理内存全拷贝一份，这极其缓慢。现代 Linux 的 `fork` 仅仅拷贝了父进程的**页表 (Page Tables)**，并将两者的物理页都标记为**只读**。只有当父或子进程试图修改某个变量时，硬件才会触发一个保护故障（Fault），内核截获后，才会真正为那一页物理内存分配副本。这就使得 `fork` 创建进程的速度极快。

### 2. `waitpid()`： 僵尸进程(Zombies)

进程调用 `exit()` 后并不会立刻消失，而是变成僵尸进程（释放了代码和数据内存，但在内核进程表里还占着一个坑位，保存着退出状态）。

- **为什么需要僵尸进程？** 保证父进程随时能查到子进程是怎么死的（正常退出还是被信号杀掉）。
- 父进程必须通过 `waitpid()` 读取这个状态，内核才会彻底抹除僵尸进程。如果父进程自己先结束了，孤儿僵尸进程会被 PID=1 的 `init`/`systemd` 进程接管并收割。

### 3. `execve()`：

`execve()` 加载一个新的可执行文件，彻底摧毁当前进程的用户态内存（代码、数据、堆、栈），并替换为新程序。**执行成功则永不返回。**

## 四、 信号 (Signals)

信号是内核向应用程序发送的小消息（ID 1~30），是 Unix 系统中最难写出正确代码的地方，因为它是**极度异步**的。

### 1. Pending 与 Blocked 位向量

内核为每个进程维护了两个隐藏的极小数据结构（各 32 bit 的位向量）：

- **`pending` 向量：** 记录内核发给了进程，但进程还没来得及处理的信号。

- **`blocked` 向量（信号掩码）：** 记录进程当前屏蔽了哪些信号。

  **触发机制：** 每次内核由于时间片轮转或中断处理完毕，准备从内核态**返回用户态**时，它都会检查 `pending & ~blocked` 是否非空。如果有信号，强制跳转到用户的 Signal Handler。

### 2. 信号是不排队的

因为 `pending` 仅仅是一个 **位向量 (Bit Vector)**

如果进程收到了 1 个 `SIGCHLD`，内核把 `pending` 的第 17 位置 1。如果在这期间，又有 3 个子进程退出了，内核再发 3 个 `SIGCHLD`，内核只能再次把第 17 位置 1（1|1=1）。

> 并发收割的问题：
>
> 如果你在 Handler 里写 `waitpid(-1, NULL, 0);`，你期望收到一个信号就收割一个子进程。但由于信号被合并了，3 个同时退出的子进程只会触发 1 次 Handler，剩下的 2 个将永远成为僵尸进程！
>
> **标准写法：** 必须在 Handler 里用 `while (waitpid(-1, NULL, WNOHANG) > 0)`。只要进了 Handler，就一口气把所有能收割的僵尸进程全部收割干净。

### 3. Async-Signal-Safety (异步信号安全)

由于 Handler 可能在主程序执行任何一行代码时突然切入，如果在 Handler 里调用了不可重入的函数，程序会瞬间死锁或崩溃。

- **禁忌：** 严禁在 Handler 中调用 `printf`、`malloc`、`free` 等标准库函数！它们内部维护了全局数据结构并使用了互斥锁。
- **正确做法：** Handler 应该尽可能简单。只调用异步信号安全的底层系统调用（如 `write`），或者仅仅置一个全局的 Flag 就立刻返回。

### 4. `volatile sig_atomic_t` 的意义

如果你在 Handler 中置了一个全局 Flag（如 `flag = 1;`），主程序用 `while(!flag)` 等待：

- 必须加 `volatile`，否则编译器 `-O2` 优化会把 `flag` 缓存在寄存器里，主程序永远在读死数据，陷入死循环。
- 必须使用 `sig_atomic_t` 类型（通常是简单的 `int`）。保证 CPU 在读写这个变量时是一条单独的汇编指令，绝不会在读写半截的微秒级瞬间被其他信号打断，造成数据撕裂。

## 五、 非局部跳转 (Nonlocal Jumps)

这是 C 语言提供的用户级异常控制流（类似 C++ 的 `try-catch`），但它极其野蛮。

- **`setjmp(jmp_buf env)`：** 将当前的调用环境（程序计数器 PC、栈指针 `%rsp`、所有被调用者保存的通用寄存器）拍个快照，存在 `env` 里。**初次调用返回 0**。
- **`longjmp(jmp_buf env, int val)`：** 无论你在多么深的函数调用栈里，它直接把 `env` 里的寄存器快照强行塞回物理 CPU 寄存器中。程序的执行流会瞬间回溯到 `setjmp` 所在的位置，并且假装 `setjmp` 返回了 `val`。

> C++ 为什么放弃 `longjmp`？**
>
> C++ 是建立在 RAII（资源获取即初始化）和面向对象基础上的语言。
>
> `longjmp` 是一种纯粹的**汇编指针强行回退**。它直接把 `%rsp` 指针拉回去了，**绝对不会调用中途所有栈帧里局部对象的析构函数。**
>
> 如果你在中间的某个函数里 `std::lock_guard` 锁住了一个 mutex，或者 `new` 了一块内存，一旦`longjmp` ，锁永远不释放（死锁），内存永远泄露。
>
> 因此，在 C++ 开发中禁止混合使用 C++ 对象与 `setjmp`/`longjmp`。
