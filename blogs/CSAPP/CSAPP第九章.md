**Virtual Memory**

**虚拟内存**

本章的主要内容为 page table、address translation、memory mapping、dynamic allocation。

## 一、 地址翻译 

一台 64 位机器，指针其实并非为64位。如在 Core i7 架构中，真实的**虚拟地址 (VA)** 只用了 48 位，**物理地址 (PA)** 只用了 52 位。

### 1. 虚拟地址的 $9+9+9+9+12$

假设页面大小为 4KB ($2^{12}$ 字节)，那么地址的低 12 位就是**页偏移 (VPO / PPO)**。

剩下的 36 位虚拟页号 (VPN)，被极其精妙地等分成了 4 段，每段 9 位，对应 **4 级页表**：

- **PGD (Page Global Directory)：** 1 级页表，由 VPN 1 寻址。
- **PUD (Page Upper Directory)：** 2 级页表，由 VPN 2 寻址。
- **PMD (Page Middle Directory)：** 3 级页表，由 VPN 3 寻址。
- **PTE (Page Table Entry)：** 4 级页表，由 VPN 4 寻址，最终指出物理页号 (PPN)。

> 为什么每级刚好是 9 位？**
>
> 一个页表项 (PTE) 的大小是 8 字节。一个 4KB 的物理页刚好能装下 $4096 / 8 = 512$ 个 PTE。
>
> 而 $512 = 2^9$，这意味着，使用 9 位地址刚好能完美索引一整页的 PTE。这种对齐设计使得操作系统在分配页表时，直接分配一个标准物理页即可，不浪费内存。

### 2. PTE 的控制位 (Control Bits)

页表项不仅仅存放物理地址，它的低 12 位为控制信号：

- **P (Present)：** 物理页是否在内存中（0 表示引发缺页异常）。
- **R/W (Read/Write)：** 是只读还是可读写。
- **U/S (User/Supervisor)：** 用户态能否访问（防止用户代码读写内核空间）。
- **D (Dirty)：** 脏位。此页是否被写（决定换出到磁盘时是否需要真写）。
- **NX (No Execute)：** 指示该页数据是否允许被当成 CPU 指令执行。

> **(NX Bit)解决缓冲区溢出**
>
> 早期可以通过缓冲区溢出，把恶意机器码注入到栈上，然后把返回地址指向栈顶来执行。后来 AMD/Intel 引入了 **NX 位**，将栈内存页的 PTE 设为 `NX = 1`。从此以后哪怕成功跳到了栈上，CPU 一接受 NX 位，立马抛出异常，使栈溢出执行攻击无法进行。

## 二、 缺页异常 (Page Fault) 

当硬件 MMU 查页表发现 `P=0` 时，硬件电路会彻底放弃，触发 Page Fault，进入 Linux 内核的 `do_page_fault` 函数。整个过程极其复杂且严谨：

### 1. CR2 寄存器

硬件在触发异常的瞬间，会把那个导致缺页的**非法虚拟地址**，死死地锁在 CPU 的 **`CR2` 寄存器**中。内核第一步就是读 `CR2`。

### 2. 查阅进程红黑树 (vm_area_struct)

内核拿着 `CR2` 里的地址，去当前进程的 `mm_struct` 里找 `vm_area_struct`（VMA）链表。由于 VMA 可能很多，Linux 实际上用了一棵**红黑树**来加速查找。

- **场景 A（越界）：** 地址根本不在任何 VMA 树节点内。内核判定野指针，抛出 `SIGSEGV` (段错误)。
- **场景 B（越权）：** 地址在 VMA 内，但该 VMA 是只读的，而你的操作是 `write`。内核判定越权，抛出 `SIGSEGV`。
- **场景 C（合法缺页）：** 地址合法且权限对，只是没加载进物理内存。进入下一步。

### 3. 按需调页 (Demand Paging)

内核挑选一个物理页（如果满了，用 LRU 算法把别人踢进 Swap 分区磁盘），把磁盘里对应的数据拉进来，修改 PTE 的 `P=1`，然后**将 PC 指针拨回原处，重新执行那句引发异常的汇编指令**。

## 三、 `mmap` 与共享库

**共享库并不是以“共享对象 (MAP_SHARED)”映射的，而是以“私有对象 (MAP_PRIVATE)”映射的**

### 1. MAP_PRIVATE 

当一千个进程链接 `libc.so` 时，它们页表里的代码段（`.text`）和数据段（`.data`）对应的物理页全是同一组。

但在 VMA 中，它们被标记为 **MAP_PRIVATE**。内核把这些物理页的 PTE 全部强制设为**只读 (Read-Only)**。

- **只读代码：** 物理内存只消耗 1 份。
- **写入数据：** 进程 A 想要修改 `libc.so` 的 GOT 表（全局偏移表）或者里面的全局变量。它一执行写指令，硬件报警（写入只读页）。内核截获异常，发现是 MAP_PRIVATE 的页，于是**在物理内存里克隆出一个新页**，把旧数据拷过去，把进程 A 的 PTE 指向新页，并设为可写。

## 四、 动态内存分配 (Dynamic Memory Allocation) 

C 语言的 `malloc` 在底层不是系统调用，它是运行在用户态的复杂算法库（分配器 Allocator）。

### 1. 隐式空闲链表 (Implicit Free List)

堆是一个连续的字节数组，被切分成各种大小的块 (Block)。每个块的结构如下：

- **Header（头部，4 或 8 字节）：** 记录块大小 (Size) 和分配状态 (Allocated Bit)。
- **Payload（有效载荷）：** 真正返回给用户的指针区域。
- **Padding（填充）：** 为了满足 8 字节或 16 字节对齐。
- **Footer（尾部边界标记，4 或 8 字节）：** 内容与 Header 完全一致，用于在释放时快速定位上一个块的结尾。

> ** 去掉 Footer榨取空间 **
>
> 在 Malloc Lab 中要想拿满分，不能老老实实给每个分配的块都加 Footer，太浪费空间了。
>
> ** 因为块大小必须是 8 的倍数，所以 Header 记录大小的最低 3 个 bit 永远是 0。我们可以把最低位（bit 0）用来记录当前块是否分配（`a/f`），**把次低位（bit 1）用来记录上一个物理块是否分配。**
>
> 如果当前块看一眼自己的 bit 1 发现前面的块已经分配了，那前面的块根本不可能参与合并 (Coalescing)，所以我根本不需要读取前面块的 Footer！**故：只有空闲块才需要 Footer。已分配块的 Footer 可以直接被砍掉，腾出空间给用户。**

### 2. 合并的 Cases (Coalescing)

当你调用 `free(p)` 时，分配器必须检查相邻的物理块并合并，以消除外部碎片：

- **Case 1：** 前后块都已分配。不合并，直接把自己的 Header 的 bit 0 清零。
- **Case 2：** 前块分配，后块空闲。利用自己的 Header 找到后块的 Header，合并成一个大块，更新 Header 和 Footer。
- **Case 3：** 前块空闲，后块分配。通过自己前 4 个字节（也就是前块的 Footer）读出前块的 Size，指针回退，合并前块，更新 Header 和 Footer。
- **Case 4：** 前后皆空闲。同时利用前块的 Footer 和后块的 Header，把三者融为一体。

### 3. 分离适配 (Segregated Fits) 

隐式链表遍历太慢，显式链表（在空闲块里存 `next/prev` 指针）又容易产生内存碎片。现代 C 标准库（如 Glibc）采用**分离适配**：

- 维护一个指针数组，每个指针指向一个按大小分类的显式双向空闲链表（如 16-31B，32-63B，64-127B）。
- **分配逻辑：** 如果用户 `malloc(20)`，直接去 16-31B 的链表里拿。如果有，以 $O(1)$ 速度返回；如果没有，去 32-63B 的链表里拿一个块，切成 24B（给用户）和剩余的 8B（扔进更小的链表），返回。
- **红黑树的引入：** 在 Glibc 中，当分配极大的内存块（Large Bins）时，由于大小差异悬殊，链表遍历依然慢，此时才会对巨大的空闲块启用额外的内部数据结构（如 Tries 或近似红黑树）来维护。

## 五、 C 语言的垃圾收集 (Garbage Collection)

 C/C++ 也是可以有垃圾收集器的（如 Boehm GC），书中也稍微提及了保守的 Mark & Sweep（标记-清除）算法。

- **瓶颈：** C 语言没有类型元数据（Metadata）。GC 扫描内存时，看到一个 8 字节的数据 `0x00007fffaaaabbbb`，它根本不知道这是一个长整型数字，还是一个真实的物理指针。
- **妥协：** C 语言的 GC 只能做**保守估计**。只要这个 8 字节的数据“看起来像”是一个指向已分配堆块范围内的合法地址，GC 就为了稳妥，宁愿把这个堆块当成存活的（Reachable），绝不敢回收它。这就是为什么 C 的 GC 永远不如 Java/Go 的 GC 高效且干净。
