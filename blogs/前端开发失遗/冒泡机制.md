
# 事件委托：当事件冒泡不足以解决问题时

随着前端应用规模的扩大，管理用户交互变得愈发重要。为每一个可交互元素单独绑定事件监听器是一种糟糕的实践，这会导致代码逻辑混乱、内存消耗激增以及性能瓶颈。这就是**事件委托 (Event Delegation)** 发挥作用的场景。

现代 Web 应用的交互建立在 DOM 及其事件系统之上。了解事件传播机制与委托模式，是构建高性能前端（无论是原生 JS 还是基于 React 等框架）的必须。

* 避免成百上千个独立监听器消耗 CPU 和内存。事件委托通过集中处理降低了系统开销。
* 统一的入口让代码更简洁、更易于调试。
* 即使 DOM 是后期动态渲染插入的（例如通过 API 拉取数据后渲染的列表），委托监听依然有效。

---

##  DOM 事件传播机制 (Event Propagation)

在深入委托之前，必须厘清事件在 DOM 树中的流转过程。事件传播包含三个明确的阶段：

1. **捕获阶段 (Capturing Phase)**：事件从 `window` 级别向下穿透 DOM 树，直到到达目标元素的父级。通过向 `addEventListener` 传入第三个参数 `{ capture: true }` 或 `true`，可以在此阶段拦截事件。
2. **目标阶段 (Target Phase)**：事件精准到达被触发的元素本身。绑定在该元素上的所有监听器在此刻执行。
3. **冒泡阶段 (Bubbling Phase)**：事件如同水泡一样，从目标元素向上传递，经过父级、祖父级直到 `window`。**这是绝大多数事件监听器默认生效的阶段**。

### 事件流演示

假设有如下 DOM 结构：

```html
<div id="grandparent">
  <div id="parent">
    <button id="child">Click Me</button>
  </div>
</div>
```

点击 `<button id="child">` 时的事件流向：

- **捕获**：`window` -> `document` -> `<html>` -> `<body>` -> `#grandparent` -> `#parent`
- **目标**：`#child`
- **冒泡**：`#child` -> `#parent` -> `#grandparent` -> `<body>` -> `<html>` -> `document` -> `window`

##  事件委托

事件委托的核心是：**不把监听器绑定在具体的子元素上，而是绑定在它们共同的祖先元素上。** 利用事件冒泡机制，当子元素触发事件时，父元素的监听器会捕获该事件，并通过判断事件源来决定后续逻辑。

### 对比分析：常规绑定 与 事件委托

如果有一个包含多个 `<li>` 的列表：

HTML

```
<ul id="myList">
  <li>Item 1</li>
  <li>Item 2</li>
  <li>Item 3</li>
</ul>
```

**反例（逐个绑定）：**

JavaScript

```
const listItems = document.querySelectorAll('#myList li');
listItems.forEach(item => {
  item.addEventListener('click', (event) => {
    console.log(`Clicked on: ${event.target.textContent}`);
  });
});
```

**正确处理（事件委托）：**

JavaScript

```
const myList = document.getElementById('myList');
myList.addEventListener('click', (event) => {
  // 精准拦截目标元素
  if (event.target.tagName === 'LI') {
    console.log(`Clicked on: ${event.target.textContent}`);
  }
});
```

在这种模式下，无论未来向 `<ul>` 中动态推入多少个新的 `<li>`，都不需要重新绑定监听器。

##  事件委托的常见问题

### 1. `event.target` 与 `event.currentTarget`

这是最容易导致 Bug 的混淆点：

- **`event.target`**：**触发事件的根本**。在上述例子中，即使监听器绑在 `<ul>` 上，只要点的是具体的 `<li>`，`event.target` 就是那个 `<li>`。
- **`event.currentTarget`**：**监听器实际绑定的元素**。在上述委托场景中，`event.currentTarget` 永远指向 `#myList` (即 `<ul>`)。

**最佳实践**：在委托逻辑中，使用 `event.target` 来匹配和操作具体的交互节点；使用 `event.currentTarget` 来进行容器级别的状态管理或监听器卸载。

### 2. 滥用阻止传播机制

- `event.stopPropagation()`：阻止事件继续向上冒泡或向下捕获。
- `event.stopImmediatePropagation()`：不仅阻止 DOM 树的传播，还直接阻断挂载在当前元素上的其他同类型监听器。

**注**：如果底层组件的某人习惯性地写了 `e.stopPropagation()`，会导致上层的事件委托全部失效。对于需要全局逻辑映射或组件通讯的前端项目而言，这往往是致命的。除非有极其特殊的边界需求，否则**应尽量避免阻断事件流**，而是通过状态判断来控制执行边界。

##  Web Components 与 Shadow DOM 的事件穿透

Shadow DOM 为组件提供了内部结构和样式的隔离，但这直接改变了事件传播的规则：

- **事件重定向 (Events re-targeting)**：当事件从 Shadow DOM 内部冒泡到外部（Light DOM）时，浏览器出于封装安全性，会将 `event.target` 强制重置为 **Shadow Host**（即自定义元素本身）。外部无法探知组件内部触发事件的具体细节节点。
- **`composed` 标志**：只有 `composed: true` 的事件（如 `click`, `keydown`）才能穿透 Shadow 边界进入 Light DOM。像 `focus` 或 `blur` (`composed: false`) 会被永远锁在 Shadow DOM 内部。
- **自定义事件与 `bubbles`**：在 Shadow DOM 内部派发自定义事件时，必须显式声明 `bubbles: true` 且 `composed: true`，否则外部的委托监听器无法捕获。

##  无法冒泡的事件及替代方案

绝大多数 UI 事件都会冒泡，但以下几个核心事件是例外：

- `focus` / `blur` (元素获得/失去焦点)
- `mouseenter` / `mouseleave` (鼠标移入/移出)

**解决方案**：

无法冒泡意味着无法直接使用传统的事件委托。但浏览器提供了对标的替代方案：

- 对于焦点管理：使用 **`focusin`** 和 **`focusout`** 代替，它们原生支持冒泡。
- 对于鼠标悬停：使用 **`mouseover`** 和 **`mouseout`**。

JavaScript

```
const form = document.getElementById('myForm'); 
// 利用 focusin 实现表单内部所有 input 的委托校验
form.addEventListener('focusin', (event) => {
  if (event.target.tagName === 'INPUT') {
    event.target.classList.add('active-field');
  }
});
```
